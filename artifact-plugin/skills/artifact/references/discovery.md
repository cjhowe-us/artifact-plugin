# discovery

The artifact plugin builds a registry at session start. It records every installed scheme, storage, and
artifact-template — never loading bodies.

## Registry location

`$ARTIFACT_CACHE_DIR/registry.json`. Platform paths via `artifactlib.xdg`:

- Linux: `${XDG_CACHE_HOME:-$HOME/.cache}/artifact/registry.json`
- macOS: `~/Library/Caches/artifact/registry.json`
- Windows: `%LOCALAPPDATA%\artifact\cache\registry.json`

Regenerated by:

- `hooks/sessionstart-discover.py` on session open (declared in `hooks/hooks.json`).
- `scripts/discover.py` on demand (after installing a new plugin mid-session).

## Registry schema

```json
{
  "generated_at": "2026-04-18T10:15:00Z",
  "entries": [
    {
      "entry_type": "artifact-scheme" | "artifact-storage" | "artifact-template" | "workflow",
      "scope": "override" | "workspace" | "user" | "plugin",
      "path": "/abs/path/to/scheme.toml",
      "name": "<name>",
      "scheme": "<scheme name>",
      "kind": "vertex" | "edge" | "metadata",
      "description": "...",
      "contract_version": 1,
      "storages": [...]              // for artifact-scheme entries: [[storage]] adapter list
    }
  ],
  "warnings": []                      // consistency issues found at discovery time
}
```

## Scope precedence

Higher precedence wins when names collide:

| Scope     | Precedence | Location                                              |
|-----------|-----------:|-------------------------------------------------------|
| override  | 1 (top)    | `$CWD/.artifact-override/<bucket>/<name>/`            |
| workspace | 2          | `$REPO/.claude/<bucket>/<name>/` (committed to repo)  |
| user      | 3          | `~/.claude/<bucket>/<name>/`                          |
| plugin    | 4 (bot.)   | `<installed-plugin>/<bucket>/<name>/`                 |

`<bucket>` ∈ `artifact-schemes`, `artifact-storage`, `artifact-templates`. Symlinks followed.

## What discovery reads

- For schemes: `scheme.toml` only. The `scheme.py` is imported lazily by the mediator when a subcommand runs (via
  `artifactlib.scheme.load_scheme`).
- For storages: `storage.toml` only. `storage.py` loaded lazily.
- For templates: body filename (must contain `.jinja.`) + companion `<stem>.content.toml`.
- For workflows (Claude-Code skills under `skills/workflows/`): YAML frontmatter of `SKILL.md`.

Bodies never read at discovery time — fast session start, minimal context cost.

## Consistency checks

Discovery cross-references each storage's `backs_schemes` against every scheme's `[[storage]]` list. Mismatches surface
in the `warnings` array but do not fail discovery.

## Dependency probe

`hooks/sessionstart-discover.py` probes `pydantic`, `tomlkit`, `jinja2` imports before running discovery. Missing deps →
a one-line hint on stderr, exit 0 (non-fatal).

## Lazy-load flow

1. `/artifact` invoked → skill reads `registry.json` (small, fast).
2. Skill interprets user intent against names + kinds.
3. Run-provider is invoked with a specific scheme/URI.
4. `artifactlib.provider` loads that scheme's `scheme.py` + the resolved storage's `storage.py`.
5. Cross-scheme subcommands (e.g. `artifact-template.instantiate`) recurse through the same path.

## Debugging

```bash
# Show the current registry
python3 -c 'import json, pathlib, os; print(json.dumps(
  json.loads(pathlib.Path(os.environ.get("ARTIFACT_CACHE_DIR", pathlib.Path.home() / "Library/Caches/artifact")) / "registry.json").read_text())
  if False else "")'
cat "$ARTIFACT_CACHE_DIR/registry.json"   # simpler

# Rebuild now
python3 <plugin>/scripts/discover.py

# Check precedence resolution for a name
python3 -c '
import json, os
r = json.loads(open(f"{os.environ[\"ARTIFACT_CACHE_DIR\"]}/registry.json").read())
print([e for e in r["entries"] if e.get("name") == "<name>"])
'
```
